Arms & Treatment: Crossover design — LY-G (LY900018 then Glucagon) and G-LY (Glucagon then LY900018). Both arms follow the same visit schedule; arm assignment determines treatment sequence. 2 arms × 5 epochs = 10 StudyCells.   Sub-timelines: Two additional sub-timelines exist: Hypoinduction (5 time-points for insulin clamp procedure) and Treatment Phase (15 intra-day time-points: 5min, 10min, 15min… 240min post-dose). These are shown separately in the Data Tables tab.   BCs: 35 unique BiomedicalConcepts — all SDTM Dataset Specialization references. Clinical Lab Tests links to 25 individual lab BCs. BC×N = number of BC links on that activity.

Architecture: USDM + COSMoS + ODM 2.0 + CRF Specialization

P21 MDR context: SQL backend, Handsontable grid views, ODM 1.3 baseline

Five-Layer Model

1
USDM — Protocol & Schedule of Activities
Owns the SoA. ScheduledActivityInstance (SAI) is the pivot: links one Encounter + Epoch + many Activities. SAI.name (V1, V1.1…) is the SoA column header; timing objects give the Day row. next_sai_id is the timeline linked list; sub_timeline_id enables branching. The Handsontable SoA pivots usdm_sai_activity.
USDM v4.0ICH M11SAI = visit node
2
COSMoS BC Catalog — Leaf Biomedical Concepts
Specific assessable concepts: Respiratory Rate (C3826), Body Temperature (C25206), Systolic BP (C25298), Diastolic BP (C25299), Heart Rate (C49677), QTc Interval (C117240), etc. “Vital Signs” is a bc_categories search tag, not a BC. One Activity links to multiple BCs via activity_bc.
COSMoSNCIt leaf BCsLOINC/SNOMED

COSMoS CRF Specialization (Draft) — Pre-Built Form Metadata
35-column schema: question_text, dataType, length, codelist, value lists, mandatory, SDTM annotations. Maps 1:1 to ODM 2.0 ItemGroupDef/ItemDef. AE, CM, DM, DS, EC in machine-readable format. VS/EG exist on CDISC eCRF Portal (member-only, CDASH) but not yet in COSMoS format. LB absent from both. Standard forms are importable.

ItemGroupDef[Type=Concept] per leaf BC. Nested IGs: IG.VS_PANEL → IG.VS.BP → IG.VS.BP.SYSBP + IG.VS.BP.DIABP. Coding element carries same NCIt/LOINC as COSMoS — zero translation. ValueListDef + WhereClauseDef = SDTM VLM pattern. Incremental upgrade of P21’s existing ODM layer.

5
SDTM Dataset Specialization — Submission Layer
Maps COSMoS BCs to SDTM via VLM and WhereClause. sdtm_target_variable in CRF Spec closes the loop. Generates Define-XML v2.1 automatically.
SDTM Dataset SpecDefine-XML 2.1
Arms & StudyCells: USDM represents arm differentiation via StudyCells (Arm × Epoch → StudyElement). When all arms share one schedule, a single SoA grid applies with an arm note. If arms diverge (different visit windows or extra procedures), USDM uses SAI.sub_timeline_id to branch into arm-specific sub-timelines. In this study all sub_timeline_ids are null — one shared schedule, three arms differing only in IMP dose.
P21 Handsontable fit: The SoA is static HTML rendered server-side from USDM data. The Data Tables tab shows each SQL table as a live Handsontable grid. Standard BC form metadata arrives pre-built from COSMoS CRF Specialization — imported once, reused across studies.
BcSurrogate: When no published SDTM Dataset Specialization or COSMoS BC exists, USDM uses a BcSurrogate — a lightweight placeholder with name, label, and reference="None set". Activities reference surrogates via bcSurrogateIds alongside any full biomedicalConceptIds. In this study: Clinical Lab Tests has 25 full BCs + 16 surrogates (LFTs, RBC indices, insulin etc.). Clinical Serology Tests has 4 surrogates only. Once CDISC publishes a DS for a surrogate concept, the surrogate is replaced with a full BC.
CRF Spec gap: VS (112 rows, 13 BCs), EG (243 rows, 24 BCs), and LB (976 rows, 77 BCs) are fully covered in the CRF Specialization XLSX. The publicly accessible GitHub CSV is an older export — use the XLSX from the COSMoS export/ folder. For LY900018: 32/35 BCs have direct CRF Spec matches. Lymphocytes (C51949) is the only LY900018 BC genuinely absent.

CRF Specialization → P21 Form Generation

The COSMoS CRF Specialization provides a flat 35-column schema that can be directly transformed into P21's three-level form hierarchy. The mapping is deterministic — no authoring required for standard domains.

CRF Specialization → P21 / ODM 2.0 Hierarchy
CRF Spec field(s)P21 levelODM 2.0 equivalentNotes
domainForm (top level)ItemGroupDef[Type=Form]One form per SDTM domain (AE, CM, DM…). SDTM target domains assigned as attribute.
crf_group_id, short_nameSectionItemGroupDef[Type=Section]e.g. AE_DENORMALIZED, CMFREE_NORMALIZED. Mandatory=No, Repeating=No by default.
crf_item, question_textQuestion / ItemItemDef + ItemRefOrder from order_number. Items ending in RESU/RES flagged as VLM targets.
data_type, length, significant_digitsQuestion attributesItemDef/@DataType, @Length, @SignificantDigitsdecimal with significant_digits → float in P21. length=200 implies text, digits dropped.
codelist (NCI C-code), value_list, value_display_listCodelist / TermsCodeList + EnumeratedItemSub-codelists generated per field. Unit sub-codelists identified by items ending in RESU.
prepopulated_termAssigned valueNo ODM 2.0 equivalentTriggers EQ WhereClause for VLM. Included as CodeList subset.
sdtm_target_variableSDTM targetAlias/annotationDomain-qualified in P21: AE.AETERM. Multi-target comma-separated.
VLM note: WhereClause generation from CRF Specialization is appropriate only for normalized sources (implementation_option = 'Normalized'). Denormalized forms use one row per concept — no VLM needed. WhereClause comparator is IN when value_list is populated, EQ when prepopulated_term is set.

Activity → Form Link via bc_id

Because the CRF Specialization carries a bc_id matching the USDM BiomedicalConcept code, the bridge from protocol activity to generated form is fully traceable:

-- USDM Activity → CRF Specialization → P21 Form
SELECT a.id, a.name AS activity,
       b.bc_code, b.ref_id AS sdtm_ds,
       c.domain AS form_id,
       c.crf_group_id AS section_id,
       c.crf_item AS question_id,
       c.question_text
FROM usdm_activity a
JOIN activity_bc b ON a.id = b.activity_id
JOIN cosmos_crf_item c ON b.bc_code = c.bc_id
-- Result: for each activity, the form questions that collect its BCs
-- Coverage limited to domains in CRF Spec (AE/CM/DM/DS/EC today)

Form Library Link (Higher-Level Alternative)

Where a P21 MDR already has a form library, the Activity→Form relationship does not need to go through the granular BC→CRF Spec chain. A direct activity_form junction table can be maintained at the activity level — auto-populated where BC links exist, manually maintained elsewhere.

-- Direct activity-to-form junction (higher-level merge)
CREATE TABLE activity_form (
  activity_id   VARCHAR, -- USDM Activity ID
  form_id       VARCHAR, -- P21 Form ID (e.g. AE, VS, LB)
  form_name     VARCHAR, -- Human-readable form name
  link_source   VARCHAR -- 'crf_spec' | 'manual' | 'library'
);
BC-level merge
Granular, traceable. activity_bc → cosmos_crf_item → form/section/question. Shows which specific BCs a form collects. Requires BC links and published CRF Specialization.
Form library merge
Pragmatic, works without BC links. activity_form junction auto-populated where possible, manually maintained for surrogates and unpublished domains. Editable in Handsontable.
Coverage caveat: The activity→form link only works where (a) the activity has biomedicalConceptIds or bcSurrogateIds, (b) the BC has a CRF Specialization entry, and (c) a BcSurrogate has been promoted to a full BC. In LY900018: Informed Consent, Concomitant Meds, and AE/DS-type activities can auto-generate forms from the machine-readable CRF Spec today. VS and EG have eCRF Portal templates (member-only, CDASH) — usable with format conversion. Clinical Lab Tests requires LB, which is absent from both the CRF Spec draft and the eCRF Portal COSMoS format.

SDTM Trial Summary Domains from USDM SoA

The USDM SoA provides all the data required to generate the five SDTM Trial Design domains (TA, TE, TV, TI, TS) directly — no manual SDTM authoring needed for these domains. The mapping is defined in SDTM IG 3.4 and the USDM-to-SDTM Mapping specification.

TSSEQ (SDTM IG 3.4 §5.5): TSSEQ is a sequence number within each TSPARMCD, not a global counter. If a parameter has multiple values (e.g. OBJSEC has 3 secondary objectives, TRT has 2 treatments), they get TSSEQ 1, 2, 3 within that parameter. TSGRPID ties related parameters together across rows — e.g. TRT and DOSE records for the same treatment arm share a TSGRPID value.
